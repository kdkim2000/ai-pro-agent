## 1. 작업 개요

### 1.1 목적

프로그램 개발 전주기 지원 AI Agent의 **설계도**를 완성하는 단계.

## 3. 발생 이슈 및 해결

### Issue 1: BaseSkill 초기화 — 단위 테스트 환경 커넥터 오류

| 항목 | 내용 |
|------|------|
| **증상** | `BaseSkill.__init__` 호출 시 `AiProChatModel()` 등 실 커넥터 초기화 → 사내망 없는 로컬 환경에서 Import 또는 연결 오류 가능 |
| **원인** | T1-3_GUIDE.md의 `BaseSkill.__init__`는 항상 실 커넥터를 생성함. 단위 테스트도 동일 코드 실행 |
| **해결** | `USE_MOCK_CONNECTORS` 환경변수 확인 로직 추가: `true`일 때 `self._llm = None`, `self._connectors = None` 처리. 테스트 파일 최상단에서 `os.environ["USE_MOCK_CONNECTORS"] = "true"` 설정 |
| **교훈** | 가이드 코드를 그대로 반영할 때도 **테스트 가능성(testability)** 을 항상 검토. Mock 전환 인터페이스를 초기부터 설계하는 것이 중요 |

### Issue 2: `AgentState` 중복 정의 (scaffold.py vs state.py)

| 항목 | 내용 |
|------|------|
| **증상** | T1-1에서 `scaffold.py`에 단순 `AgentState`가 정의되어 있고, T1-3에서 `state.py`에 완전한 `AgentState`를 신규 정의 → 같은 이름이 두 곳에 존재 |
| **원인** | T1-1은 뼈대(Scaffold), T1-3은 완전한 설계. 기존 테스트(test_scaffold.py)가 scaffold 의존 |
| **해결** | `scaffold.py` 유지, `src/agent/__init__.py`에서 `state.py` 기준으로 export 교체. 기존 `test_scaffold.py`는 `scaffold.py`를 직접 import하므로 영향 없음 |
| **교훈** | 점진적 리팩토링 시 기존 파일을 삭제하지 않고 **export 레이어(\_\_init\_\_.py)만 교체**하면 하위 호환이 유지됨 |

### Issue 3: gate_rules.yaml 포맷 불일치

| 항목 | 내용 |
|------|------|
| **증상** | T1-1에서 생성된 `gate_rules.yaml`은 `gates.requirement.required_fields` 형태의 중첩 구조. T1-3 GateEngine은 `gates[].check_type` 리스트 형태 사용 |
| **원인** | T1-1은 구조 예시, T1-3은 실제 Rule Engine이 파싱할 포맷 |
| **해결** | `gate_rules.yaml` 전체 재작성 — `version: "1.0.0"` + `gates: [{ name, description, required, check_type, field, ... }]` 리스트 구조로 통일 |
| **교훈** | Config 파일은 **사용 코드가 결정하는 포맷**으로 작성해야 함. 예시 포맷과 실제 파싱 포맷을 일치시킬 것 |

## 5. 설계 결정 사항

### 5.1 scaffold.py 유지 결정

T1-1 `scaffold.py`의 `AgentState`(단순 TypedDict)와 T1-3 `state.py`의 `AgentState`(완전한 TypedDict)가 같은 이름으로 공존한다.  
`__init__.py`에서 `state.py` 기준으로 export를 교체하여 새 코드는 완전한 스키마를 사용하고, `test_scaffold.py`는 scaffold를 직접 import하므로 영향을 받지 않는다.

### 5.2 Skill Stub 설계 전략

T3에서 8개 Skill을 개별 구현할 때 **파일명과 클래스명만 유지**하면 노드 함수(`nodes.py`)를 수정하지 않아도 된다. `nodes.py`는 `from src.skills.s01_requirement import RequirementSkill` 방식으로 지연 임포트하므로, T3에서 해당 파일만 교체하면 DAG 전체가 자동으로 업그레이드된다.

### 5.3 GateEngine — LLM 없는 설계

가이드 설계 원칙: "게이트 판별은 LLM에 위임하지 않고 확정적 조건으로 처리".  
`check_type` 5종(`field_exists`, `min_length`, `list_min_count`, `bool_flag`, `all_items_exist`)으로 YAML 규칙을 코드 없이 확장 가능하다.  
새로운 게이트 조건은 `gate_rules.yaml`에 항목 추가만으로 대응한다.

## 7. 학습 포인트 (VIBE 코딩 패턴)

### 7.1 의존성 순서 기반 구현

- 파일 간 의존성 그래프를 먼저 파악하고, **의존성이 없는 것부터 생성**
- `state.py` → `engine.py` → `base.py` → `nodes.py` → `workflow.py` 순서
- 순서를 지키면 중간에 ImportError 없이 점진적으로 검증 가능

### 7.2 가이드 코드 보완

- 가이드 코드를 그대로 반영하되, **테스트 가능성** 확보를 위한 보완은 적극 적용
- `USE_MOCK_CONNECTORS` 패턴은 T1-2 ConnectorFactory에서 확립된 패턴을 BaseSkill에도 일관 적용

### 7.3 Config 기반 확장성

- GateEngine의 `check_type` 메커니즘: Python 코드 변경 없이 `gate_rules.yaml`만 수정하면 게이트 규칙 추가·변경 가능
- `gate_rules.yaml`의 `version` 필드로 GateResult에 판별 시점의 규칙 버전이 기록됨 → 감사 추적 가능

### 7.4 Stub 우선 접근법

- T1-3 단계에서 8개 Skill을 모두 실제 구현하지 않고, **인터페이스(BaseSkill)만 확정**하고 Stub으로 대체
- Stub도 올바른 `SkillResult` 구조를 반환하므로 DAG 전체 흐름을 실제와 동일하게 테스트 가능
- T3에서 Stub을 교체하는 시점에 `nodes.py`와 `workflow.py`는 수정 불필요

### 7.5 HITL 패턴 설계

- `interrupt_before` 파라미터로 **노드 진입 직전** 중단 → 노드 함수에 interrupt 코드가 없어도 됨
- `node_hitl_wait()` 팩토리로 HITL 노드를 동적 생성 → 3개 중단점이 동일한 로직을 공유하면서 이름만 다름
- `Command(resume="approved"|"rejected")` 패턴으로 외부에서 재개 가능
