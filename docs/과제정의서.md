## 과제 정의

### 대상

- **사용자** : 중공업IT파트 애플리케이션 운영·개발 담당자 54명
- **적용 환경 (실제 연동 대상 시스템)**
  - 두드림 (삼성중공업 요청관리 시스템) — CR 접수·이력 데이터 소스
  - JSM (SDS 변경관리 솔루션) — 변경 등록 및 승인 워크플로우
  - GitHub Enterprise — 형상관리·PR·코드베이스 검색 대상
  - Confluence — 개발 문서·운영 매뉴얼·산출물 저장소
  - 프로그램마스터·테이블마스터·용어사전·단어사전 (자체 개발 관리 시스템)
  - Oracle 19c 딕셔너리 — 테이블·컬럼 메타데이터 영향도 분석 소스
- **연간 약 2,615건의 CR(Change Request)이 발생하는 애플리케이션 운영 환경** (26년 상반기 실측 기준 연 환산)
- 초기(MVP)는 빈발 유형 CR부터 적용 후 단계적으로 전체 확대 예정

---

### 선정 배경

- 이벤트(CR) 수신은 두드림으로 자동화되어 있으나, 이후 요구사항 구체화·영향도 분석·산출물 작성·관리 포인트 등록은 모두 담당자 개인 경험과 기억에 의존하여 처리됨
- 각 단계별 완료 기준이 존재하지 않아 프로그램마스터·테이블마스터·용어사전 등 다수의 관리 포인트 누락이 구조적으로 반복 발생
- Confluence·GitHub Enterprise·두드림에 산재한 과거 작업 이력과 코드 변경 맥락이 체계적으로 재사용되지 못하고 매 건마다 수작업 탐색 반복
- 공수 산정과 산출물 작성이 담당자에 따라 달라 품질 편차가 크고, 신규 인력 온보딩 시 프로세스 숙지에 과도한 시간 소요

---

### Pain Point

**【P1】 관리 포인트 구조적 누락**

프로그램마스터·테이블마스터·용어사전·단어사전 등록, Oracle 딕셔너리 정합성 확인 등 단계별 필수 점검 항목이 담당자 기억에만 의존 → 누락 발생 → 운영 반영 후 결함·재작업 악순환

**【P2】 표준화 부재로 인한 품질 편차**

요구사항 구체화·공수 산정·산출물 작성 방식이 담당자마다 달라 결과물 품질 불균일, 두드림·JSM 등 시스템별 등록 양식을 매번 개인이 직접 숙지·작성해야 하는 비효율

**【P3】 과거 지식의 재사용 불가**

Confluence·GitHub·두드림에 산재한 작업 이력·코드 변경 맥락이 체계화되지 않아 유사 요청 시 매번 수작업 탐색, 숙련자 부재 시 처리 품질 급락

**【P4】 반복 부대업무의 개발 역량 잠식**

건당 요구사항 확인·영향도 분석·산출물 작성·시스템 등록·PR·배포 준비 등 비개발 업무에 추정 6~8시간 소요 → 연간 약 15,690~20,920시간 (2,615건 기준) 낭비

## 개선 계획

### 개선 방법

**전체 처리 흐름 (E2E Pipeline)**

```
① CR 접수(두드림)
② 요구사항 구체화 (GitHub·Confluence·처리 이력 RAG 검색)
③ 영향도 분석 (Oracle 딕셔너리·코드베이스 조회)
④ 공수 산정 + Task 분해 (유사 CR 이력 기반)
⑤ 산출물 초안 생성 (요구사항 분석서·영향도 분석서·테스트 정의서)
⑥ 시스템 등록 지원 (JSM·프로그램마스터·테이블마스터·용어사전·단어사전 입력 초안)
⑦ 관리 포인트 게이트 점검
⑧ 배포 준비 지원 (PR 본문·현업 테스트 요청·Release 체크리스트)
⑨ 완료 처리 및 이력 축적 → 차기 공수 산정·유사 사례 재활용
```

**Pain Point별 해결 방안**

- **【P1 대응】 게이트 기반 누락 방지**
  각 단계 완료 조건을 Config로 명시, 미충족 시 다음 단계 진행 차단 및 경고 → 마스터 등록·용어 준수·Oracle 정합성 누락을 구조적으로 제거

- **【P2 대응】 산출물 자동 생성 + 표준 템플릿 적용**
  Confluence 표준 템플릿 기반 요구사항 분석서·영향도 분석서·테스트 정의서 초안 생성 → 담당자는 검토·확정만 수행, 품질 편차 최소화

- **【P3 대응】 RAG 기반 지식 재사용 체계화**
  GitHub·Confluence·두드림 이력을 벡터 인덱싱, 유사 사례 자동 조회 → 담당자 수작업 탐색 제거, 이력 축적으로 점진적 품질 향상

- **【P4 대응】 비개발 업무 자동화**
  JSM·마스터·사전 등록 입력값 초안, PR 본문·배포 체크리스트 자동 생성 → 건당 부대업무 6~8시간 → 3~4시간 목표

**HITL(Human-in-the-Loop) 원칙 준수**

Agent는 초안 생성·점검·안내까지만 수행. 두드림·JSM 실제 등록, GitHub PR 발행, 배포 승인은 반드시 담당자가 확인 후 직접 수행하여 오자동화 리스크 차단

- **2단계** : Agent 통합 운영, 전 유형 CR 확대 적용 및 목표 수치 달성 검증
- **3단계** : MCP 기반 사내 시스템 직접 연동으로 추가 자동화 확대

## AI 기술 활용

### 활용 계획 (기술 선택 근거 포함)

> 각 기술은 특정 Pain Point 해결을 위해 선택되었으며, 단순 API 호출이 아닌 직접 구현을 통해 목적에 맞게 최적화한다.

---

#### ① RAG(Retrieval-Augmented Generation) — 【P3 해결】

**선택 근거**
: 2,615건의 CR 이력·코드·문서가 GitHub·Confluence·두드림에 산재. Fine-tuning은 데이터 갱신 비용이 크고, 순수 LLM은 사내 코드베이스 미인식 → RAG만이 최신 사내 지식을 지속 반영하면서 저비용으로 유사 사례 조회 가능

**구현 방식**

| 항목 | 내용 |
|------|------|
| 청크 전략 | 코드(함수·클래스 단위), 문서(섹션 단위), CR 이력(건 단위) 각각 최적화 |
| 검색 방식 | 하이브리드 검색 : 키워드(BM25) + 벡터(임베딩) 병행으로 검색 재현율 향상 |
| Reranker | Cross-encoder 기반 재정렬로 도메인 적합도 보정, 검색 정밀도 향상 |

---

#### ② LangGraph 기반 Stateful Agent Orchestration — 【P1·P4 해결】

**선택 근거**
: CR 처리는 8단계 조건부 분기(DB 작업 포함 여부, 신규/변경 구분 등)와 단계 간 상태 유지가 필요. 단순 Chain으로는 게이트 로직 구현 불가 → LangGraph의 DAG + State Machine이 유일하게 적합

**구현 방식**

| 항목 | 내용 |
|------|------|
| DAG Workflow | CR 유형별 Conditional Edge로 필요 노드(Skill)만 활성화 |
| State 관리 | 단계 간 컨텍스트(이전 산출물·게이트 결과) 자동 전달 |
| 게이트 노드 | 각 단계 완료 조건을 Config로 정의, 미충족 시 분기 차단 |

---

#### ③ Tool-using Agent — Skill 구성 【P2·P4 해결】

**선택 근거**
: 단순 LLM 호출로는 Oracle 딕셔너리 조회·GitHub API·Confluence 쓰기 등 외부 시스템 연동 불가. Tool Calling으로 실제 데이터를 다루는 Agent 필수

**Skill별 구성**

| Skill | 핵심 기술 | 선택 근거 | 연동 시스템 |
|-------|-----------|-----------|-------------|
| 요구사항 구체화 | RAG + LLM | 사내 이력 기반 모호성 해소, 일반 LLM 불가 | GitHub, Confluence, 두드림 |
| 영향도 분석 | Oracle 딕셔너리 API + AST 코드 분석 | 실시간 메타데이터 조회 필수, 정적 분석 한계 | GitHub, Oracle 19c |
| 공수 산정 | 유사 CR 벡터 검색 | 경험 기반 산정의 구조화·객관화 | 두드림 처리 이력 |
| 산출물 초안 | LLM + 템플릿 스키마 | Confluence 표준 양식 준수 보장 | Confluence |
| 시스템 등록 지원 | Tool Calling | JSM·마스터 API 실시간 조회·검증 | JSM, 자체 관리 시스템 |
| 관리 포인트 게이트 | Config 기반 Rule | LLM 판단에 의존하지 않는 확정적 차단 로직 | 자체 시스템, Oracle 19c |
| 배포 준비 | GitHub API | PR 본문 자동 구성, 체크리스트 실시간 검증 | GitHub Enterprise |

---

#### ④ 이력 기반 자기 개선 루프 — 【P3·P4 장기 해결】

- 완료 CR 실적 공수·산출물·재작업 여부를 구조화 저장
- 공수 산정 Skill 이력 풀(Pool) 자동 갱신 → 산정 정확도 점진적 향상
- 담당자 승인/반려 피드백을 Reranker 학습 데이터로 활용 → 검색 품질 지속 개선

### 최적화 계획

#### 단순 API 호출이 아닌 직접 구현 영역 (기술 이해 증명)

- **청크 전략 커스터마이징** : 파일 유형(코드·문서·이력)별 분할 방식을 직접 설계·구현
- **하이브리드 검색 파이프라인** : BM25 + Dense Retrieval 결합 가중치 튜닝
- **Reranker 도입** : Cross-encoder 기반 재정렬로 검색 품질 보정
- **게이트 로직 구현** : LangGraph State Node에 Config 기반 완료 조건 판별 모듈 직접 개발
- **Oracle 19c 딕셔너리 연동 모듈** : `ALL_DEPENDENCIES`·`USER_TABLES` 뷰 활용 영향도 쿼리 직접 작성

#### 성능·비용 최적화 (Token Economics)

| 최적화 항목 | 방법 |
|------------|------|
| 동적 컨텍스트 조정 | CR 복잡도에 따라 RAG Top-k·시간 윈도우·소스 Mix 자동 조절 |
| 캐시 전략 | Oracle 딕셔너리·코드베이스 반복 조회 결과 캐싱 (변경 이벤트 시 무효화) |
| 입력 최적화 | 검색 결과 관련도 하위 항목 컷오프, 프롬프트 구조화로 토큰 절감 |

#### 품질 보증 (QA)

- **골든셋 기반 회귀 테스트** : 대표 CR 유형별(신규 개발·DB 변경·단순 수정) 정답 셋 구성, Skill 업데이트 시 자동 회귀 테스트 실행, 성능 저하 감지
- **신뢰도 기반 에스컬레이션** : LLM 출력 신뢰도 낮을 시 담당자에게 불확실성 명시 고지, 에스컬레이션 케이스 누적 → 취약 시나리오 식별 및 골든셋 보강
- **게이트 조건 버전 관리** : 프로세스 변경 시 코드 수정 없이 Config만 갱신

---

#### 안전성·감사 (Safety & Auditability)

- **자동 실행 금지 원칙** : 두드림·JSM 실제 등록, GitHub PR 발행, 메일 발송 등 외부 영향 행위는 반드시 담당자 확인 후 수행 (Agent는 초안·점검까지만)
- 전 Skill 실행 이력(입력·출력·판단 근거·게이트 결과) 로그 보존, 감사 추적 가능
- 민감 정보(접속 토큰·개인정보) LLM 전달 전 마스킹 처리
